## Español

Gracias por interesarte en contribuir a un proyecto de **Codara Solutions**. Esta guía aplica por defecto a todos los repositorios de la organización, salvo que un repo específico defina su propio `CONTRIBUTING.md`.

### Antes de empezar

- Revisa los [issues abiertos](https://github.com/Codara-Solutions) por si tu idea ya está en marcha.
- Para cambios grandes, abre primero un issue describiendo la propuesta y espera comentarios antes de invertir tiempo en la implementación.
- Lee el [Código de Conducta](./CODE_OF_CONDUCT.md). Aplica en todas las interacciones.

### Flujo de trabajo

1. **Fork** del repositorio y `clone` local.
2. Crea una rama descriptiva: `feat/nombre-corto`, `fix/issue-123`, `docs/seccion`.
3. Mantén commits pequeños y con mensaje claro (preferimos [Conventional Commits](https://www.conventionalcommits.org/)).
4. Si añades funcionalidad, añade pruebas y actualiza la documentación.
5. Asegúrate de que linter, tests y build pasan en local antes de abrir el PR.
6. Abre el Pull Request rellenando la plantilla. Enlaza el issue relacionado.

### Estilo y convenciones

- Sigue el estilo del repo: linter y formateador configurados son la fuente de verdad.
- En código y comentarios, prefiere **inglés** salvo que el repo indique lo contrario.
- Mensajes de PR e issues pueden ser en español o inglés.
- Evita reformatear archivos no relacionados con tu cambio.

### Revisión

- Un mantenedor revisará el PR. Si pedimos cambios, suele ser por contexto que tú no tienes — no por falta de calidad.
- Los PRs sin actividad durante 30 días pueden cerrarse; siempre puedes reabrirlos.

### ¿Dudas?

Abre un issue de tipo *Pregunta* o escribe a [info@codarasolutions.com](mailto:info@codarasolutions.com).

## English

Thanks for your interest in contributing to a **Codara Solutions** project. This guide applies by default to all organization repositories, unless a specific repo defines its own `CONTRIBUTING.md`.

### Before you start

- Check [open issues](https://github.com/Codara-Solutions) in case your idea is already in progress.
- For larger changes, open an issue first describing the proposal and wait for feedback before investing time in the implementation.
- Read the [Code of Conduct](./CODE_OF_CONDUCT.md). It applies to all interactions.

### Workflow

1. **Fork** the repository and `clone` it locally.
2. Create a descriptive branch: `feat/short-name`, `fix/issue-123`, `docs/section`.
3. Keep commits small with clear messages (we prefer [Conventional Commits](https://www.conventionalcommits.org/)).
4. If you add functionality, add tests and update the docs.
5. Ensure linter, tests and build pass locally before opening the PR.
6. Open the Pull Request and fill in the template. Link the related issue.

### Style & conventions

- Follow the repo's style: linter and formatter are the source of truth.
- In code and comments, prefer **English** unless the repo states otherwise.
- PR and issue messages can be in Spanish or English.
- Don't reformat files unrelated to your change.

### Review

- A maintainer will review the PR. If we ask for changes, it's usually about context you don't have — not about quality.
- PRs with no activity for 30 days may be closed; you can always reopen them.

### Questions?

Open a *Question* issue or email [info@codarasolutions.com](mailto:info@codarasolutions.com).
